# 文档先行开发策略（Document-First Policy）

> 目标：降低重复开发与回归风险，让每次开发都先对齐已有认知，再落地代码。

---

## 1. 适用范围

以下任务**必须**执行“文档先行”：

- 新功能开发（feature）
- 大型 Bug 修复（影响 2+ 模块，或引发线上/核心流程问题）
- 架构调整、接口契约变更、数据结构变更

以下任务可简化：

- 单文件小修（如文案 typo、样式微调、注释补充）

---

## 2. 开发前必做（先读后改）

在任何代码改动前，至少完成：

1. 阅读 `README.md`（确认当前系统边界与运行方式）
2. 阅读 `docs/novelist-agent-requirements.md`（确认需求与验收标准）
3. 阅读与本任务直接相关的专题文档（如部署、图谱、写作流程等）
4. 检索 `docs/` 中历史问题/修复记录，避免重复踩坑

建议形成最小“前置检查清单”：

- 这个问题/功能是否已有文档结论？
- 是否已有类似实现路径？
- 本次改动是否会破坏既有契约（API/数据/流程）？

---

## 3. 开发中约束

- 先复用已有模式，再新增实现
- 变更尽量最小化，可回滚
- 若发现文档与代码不一致，优先修正文档（或在 PR 中同步修正）

---

## 4. 开发后文档回写（强制）

满足以下任一条件时，**必须更新文档**：

- 修复大型 Bug
- 新增功能
- 改动 API、配置项、运行时行为

至少更新一项：

- `README.md`（能力、配置或流程变化）
- `docs/QUICKSTART.md` / `docs/deployment-server.md`（启动或部署变化）
- 相关专题文档（领域行为变化）

并在提交说明中写清：

- 改了什么
- 为什么改
- 对应更新了哪些文档

---

## 5. 验收门禁（Definition of Done）

任务完成前必须同时满足：

1. 代码验证通过（测试 / 构建 / lint 至少覆盖改动侧）
2. 文档已更新且与代码一致
3. 变更说明包含“文档更新项”

不满足上述任一项，视为未完成。

---

## 6. 推荐执行顺序（每次任务）

1. 读文档（README + 需求 + 专题）
2. 确认历史方案与风险
3. 实施最小改动
4. 跑验证命令
5. 更新相关文档
6. 提交（中文 commit message）
